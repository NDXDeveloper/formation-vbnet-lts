# 1.6.2 La stratégie hybride VB.NET / C# en une page 🔗

La sous-section précédente s'achevait sur « la troisième voie ». La voici, **en une page**.
Le traitement complet fait l'objet du **module 10** ; la frontière sujet par sujet, de
l'**Annexe B**.

## L'idée en une phrase

Parce que VB et C# compilent vers le **même IL** et partagent le *runtime*, la **BCL** et
**NuGet**, une bibliothèque C# et un projet VB **coexistent sans friction dans une même
solution**. On garde donc VB **là où il est fort**, et on écrit (ou consomme) du **C#** pour
ce qui **sort de son périmètre** — sans choisir un langage à l'échelle du projet entier.

## Le principe : répartir les rôles

- **VB.NET** → interface (Windows Forms / WPF), logique métier, code legacy : là où vivent
  votre code et votre expertise, et où VB est productif.
- **C#** (en bibliothèque) → les briques **hors périmètre VB** ou **critiques en
  performance** : *records* à déclarer, *source generators* à écrire, code `Span`-first,
  Minimal APIs, morceaux Native AOT, clients gRPC…
- **Une seule solution**, plusieurs projets, deux langages.

```text
┌──────────────────────────────────────────────────┐
│   Une seule solution .NET  (.sln / .slnx)        │
│                                                  │
│   Présentation (WinForms / WPF)  ──►  VB.NET     │
│   Logique métier                 ──►  VB.NET     │
│   Code legacy maintenu           ──►  VB.NET     │
│   · · · · · · · · · · · · · · · · · · · · · · ·  │
│   Bibliothèque « moderne »       ──►  C#         │
│   (perf, records, generators,                    │
│    Minimal API, AOT, gRPC…)                      │
│             ▲                                    │
│             └──  référencée et consommée par VB  │
└──────────────────────────────────────────────────┘
```

## Comment ça marche, concrètement

En vue d'ensemble (les détails et l'atelier sont au module 10) :

1. une **bibliothèque de classes C#** (`.csproj`) est ajoutée à la solution ;
2. le projet VB la **référence** (référence de projet, ou paquet NuGet) ;
3. côté VB, on **appelle son API publique** comme n'importe quel type .NET — y compris les
   *records* et propriétés `init-only`, qui se **consomment** sans difficulté.

> 💡 **Le seul vrai point d'attention.** Concevez une **surface publique « VB-friendly »** :
> évitez, à la frontière, les constructions C# peu consommables depuis VB. (→ 10.4 et 10.6 ;
> aide à la lecture du C# en Annexe A.)

Deux formes d'hybride coexistent :

- **Cas courant** : une **application VB consomme une bibliothèque C#**.
- **Cas « couche entière »** (UI moderne) : **C# possède la couche** (front-end Blazor /
  MAUI) et **consomme une bibliothèque métier partagée écrite en VB**.

## Quand l'adopter — et quand s'en passer

- **Adoptez l'hybride** quand la grille (→ 1.6.1) penche **en partie** vers VB et **en
  partie** vers C#.
- **Inutile** de l'imposer si **tout** relève du cœur VB (restez en VB) ou si **tout** est
  hors périmètre (faites du C# directement). L'hybride répond à des **besoins réellement
  mixtes** — ne le sur-architecturez pas.

## Le bénéfice, en un mot

Le **meilleur des deux mondes** : les forces de VB et votre base de code existante, **sans
être bloqué** par le gel de son langage ; C# prend en charge le moderne et la performance ;
le tout dans **une solution cohérente**. Le périmètre figé de VB cesse d'être un mur pour
devenir une **répartition des rôles**.
