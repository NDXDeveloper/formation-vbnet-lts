# 1.6.1 VB.NET vs C# : quand choisir quoi (grille de décision)

La section précédente a posé la posture : la question n'est pas « VB **ou** C# ? » dans
l'absolu, mais « qu'est-ce que je garde en VB, qu'est-ce que je délègue à C# ? ». Cette
sous-section rend ce choix **opérationnel**, sous la forme d'une **grille de décision**.

## Le point de départ honnête

Rappelons l'essentiel (→ 1.1) : VB.NET et C# partagent le même *runtime*, la même
bibliothèque (BCL) et le même NuGet. **Le choix porte donc rarement sur la capacité** —
presque tout ce qu'on peut atteindre depuis l'un, on peut l'atteindre depuis l'autre. Il
porte sur le **contexte** : code existant, compétences de l'équipe, type d'application,
exigences, et horizon du logiciel.

Soyons directs sur le cas le plus simple : pour un **projet neuf, sans aucune contrainte
VB**, **C# est le choix par défaut pragmatique** en 2026 — écosystème le plus large, accès
à tous les *workloads*, dernières fonctionnalités de langage, et entraînement majoritaire des
modèles d'IA. Mais c'est un **défaut, pas un verdict** : dans des situations bien
identifiées, VB.NET reste le bon choix — parfois clairement le meilleur.

## Ce qui penche vers VB.NET

- une **base de code VB.NET existante** à maintenir ou faire évoluer ;
- une **équipe qui maîtrise VB** (atelier historiquement VB) ;
- un **scénario cœur** : Windows Forms ou WPF, bibliothèque de classes, application de
  données / métier (LOB), interopérabilité Office / COM ;
- une **migration depuis VB6** : le chemin vers VB.NET est souvent moins coûteux (syntaxe
  familière, outils de conversion ciblant VB.NET) — même si certaines équipes préfèrent
  consolider directement sur C# (→ 11.2 et 18.4) ;
- une **priorité à la stabilité et à la longévité** (logiciels métier à longue durée de
  vie) ;
- une **préférence pour une syntaxe explicite et lisible**.

## Ce qui penche vers C#

- un **projet neuf sans contrainte VB** ;
- un **workload moderne** : Blazor (web), .NET MAUI (mobile / multiplateforme), Minimal
  APIs, Native AOT, gRPC, GraphQL, microservices cloud-native ;
- le besoin d'**écrire** (et pas seulement de consommer) les **dernières nouveautés de
  langage** : *records*, filtrage de motifs avancé, constructeurs primaires, *source
  generators*, code `Span`-first orienté performance ;
- un **alignement écosystème / recrutement** : vivier de développeurs, bibliothèques,
  documentation et exemples majoritairement en C# ;
- le fait que **l'essentiel du framework et de la communauté** s'exprime en C#.

## La grille de décision

| Critère | ✅ Plutôt VB.NET | ✅ Plutôt C# |
|--------|------------------|--------------|
| **Code existant** | Base VB.NET à faire évoluer | Aucune base VB (ou base C#) |
| **Équipe** | Compétence VB confirmée | Compétence C# ou polyglotte |
| **Type d'application** | WinForms/WPF, bibliothèque, LOB/données, interop Office/COM | Web (Blazor), mobile (MAUI), microservices, cloud-native, Minimal API |
| **Migration** | Depuis VB6 (chemin souvent plus court) | Réécriture profonde déjà décidée |
| **Fonctionnalités de langage** | Le figé suffit (stabilité) | Besoin d'**écrire** les nouveautés (records, AOT, generators) |
| **Écosystème / recrutement** | Critère secondaire | Critère prioritaire |
| **Horizon du logiciel** | Durable, faible *churn* | Innovation continue |
| **Performance de pointe** | Cas courants (gains *runtime* « gratuits », → 14.6) | Code critique exploitant `Span` / AOT au niveau langage |

> 💡 **Comment lire la grille.** Aucune ligne ne « gagne » à elle seule. Pondérez selon
> votre contexte : en pratique, **le code existant et la compétence de l'équipe** pèsent
> souvent le plus lourd, devant même le type d'application.

## Une heuristique en quatre questions

Pour trancher rapidement, déroulez ces questions dans l'ordre :

1. **Existe-t-il une base VB.NET à faire évoluer ?** → *Oui* : VB.NET par défaut (sauf
   raison forte **et budgétée** de migrer, → 18.4).
2. **Sinon, est-ce un scénario cœur VB (WinForms/WPF, bibliothèque, LOB, interop Office)
   et l'équipe maîtrise-t-elle VB ?** → *Oui* : VB.NET est un excellent choix.
3. **Le projet exige-t-il un workload hors périmètre VB (Blazor, MAUI, Minimal API, AOT,
   gRPC, microservices) ?** → *Oui* : C# pour cette partie — ou **hybride** (cœur C#,
   UI / métier en VB).
4. **Projet neuf, sans contrainte VB, équipe polyglotte ?** → *Oui* : **C#** par défaut.

Réponses partagées ou besoins mixtes ? C'est le signe qu'il ne faut pas choisir globalement,
mais adopter la **troisième voie** (ci-dessous).

## Pièges à éviter

- **Réécrire par principe.** Une application VB.NET qui fonctionne sur un scénario cœur n'a
  pas à devenir C# « parce que C# est plus moderne » : une réécriture, c'est du **coût et du
  risque** (→ module 11).
- **Démarrer en VB un projet clairement hors périmètre** (Blazor, MAUI…) « par habitude » :
  vous lutteriez contre l'outillage.
- **Raisonner en tout-ou-rien.** Le cadrage binaire fait passer à côté de l'hybride.
- **Oublier le facteur IA.** En VB, le coût de développement inclut la **validation
  systématique** du code généré (→ 1.6.3).

## La troisième voie : l'hybride

Quand la grille penche **en partie** vers VB et **en partie** vers C#, vous n'êtes pas
obligé de choisir à l'échelle du projet entier. On peut **garder VB là où il est fort**
(interface, logique métier, legacy) et **déléguer à C#** ce qui est hors périmètre ou
critique en performance — le tout dans une **même solution**. C'est précisément l'objet de
la sous-section suivante. (→ 1.6.2)
